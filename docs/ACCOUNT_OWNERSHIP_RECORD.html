LX 온보딩 앱 — 계정 소유권은 누구에게 있어야 하는가

GitHub Pages(화면 배포)와 Google Sheets/Apps Script(데이터 처리) 두 축 모두, 결국 "누구의 계정으로 소유·운영할 것인가"라는 질문이 남습니다. 이 문서는 그 질문에 대해 검토한 대안과, 최종적으로 무료 개인 계정 조합을 선택한 이유를 정리합니다.

01 — 배경

"무료로 돌아간다"는 것과 "누구 명의로 돌아가는가"는 별개 문제

아키텍처 결정 기록에서 GitHub Pages + Google Apps Script 조합이 월 비용 0원으로 운영 가능하다는 점을 확인했습니다. 하지만 비용이 0원이라는 것과, 그 서비스들의 "소유권(Owner)"이 누구에게 있는지는 다른 문제입니다. 두 서비스 모두 결국 특정 계정에 소유권이 걸리게 되어 있고, 그 계정을 가진 사람이 바뀌거나 퇴사하면 무슨 일이 일어나는지를 먼저 확인해야 안전하게 운영할 수 있습니다.

02 — GitHub Pages

사이트를 계속 배포하려면 결국 Owner가 있어야 한다

정적 사이트 자체는 무료로 호스팅되지만, 그 저장소를 관리·수정·재배포할 권한을 가진 사람 — 즉 Owner — 은 반드시 존재해야 합니다. Owner가 되려면 GitHub 계정이 있어야 하므로, "담당자 중 누군가는 GitHub 계정을 만들어 Owner로 등록되어야 한다"는 조건이 항상 따라붙습니다.

Owner가 1명뿐이면?
GitHub은 조직(Organization)에 Owner가 단 1명일 경우, 그 사람이 조직을 탈퇴하는 것 자체를 막습니다 — 조직이 관리자 없는 상태가 되는 걸 방지하기 위한 안전장치입니다. 반대로 말하면, 그 1명의 계정에 문제가 생기거나(정지·분실) 그 사람이 급하게 인수인계 없이 이탈해야 하는 상황에서는 조직 자체가 잠길 위험이 있습니다.
완화: 개인 계정이 아닌 Organization으로 저장소를 운영하고, Owner를 최소 2명 이상 유지합니다. Owner는 설립자 여부와 무관하게 완전히 동등한 권한을 가지므로, 담당자가 바뀌어도 남은 Owner가 새 담당자를 추가하고 이전 담당자를 내보내는 정상적인 인계가 가능합니다.
개인 계정 소유 시 리스크
만약 어느 한 담당자의 개인 GitHub 계정 아래에 저장소를 만들면, 사이트 전체가 그 사람 개인 소유물이 됩니다. 담당자가 바뀔 때마다 저장소 이전(transfer)이라는 별도 작업이 필요하고, 이전에 동의하지 않거나 연락이 닿지 않으면 회사 자산이 개인에게 묶여버립니다.
완화: 처음부터 개인 계정이 아닌 회사(프로젝트) 소유의 Organization으로 시작해, "누구 개인 것"이 아니라 "조직의 것"이라는 소유 구조를 명확히 합니다.
03 — Google Sheets / Apps Script

유료(회사 계정) vs 무료(개인 Gmail) — 왜 무료 계정을 택했는가

데이터 저장(Sheets)과 자동화(Apps Script) 역시 결국 특정 Google 계정 소유로 귀속됩니다. "회사 정식 계정(Google Workspace)"과 "개인 무료 Gmail 계정" 중 하나를 골라야 했고, 각각의 한계를 검토한 끝에 무료 계정을 선택했습니다.

계정 종류별 비교

구분회사 정식 계정 (Workspace)개인 Gmail 계정 (무료)
신규 발급 회사(IT 부서) 차원의 공식 계정 신설 필요 — 비공식 개인 프로젝트 신분으로는 요청·승인 자체가 어려움 누구나 즉시 무료로 생성 가능
메일 발송 시 신뢰도 회사 공식 도메인으로 발송되어 문제 없음 "보낸사람"이 @gmail.com으로 그대로 노출되어 신뢰도 문제 발생
소통 창구 기존 사내 메일 체계와 일원화됨 회사 공식 메일과 별도 창구가 생겨 이중 관리 필요
비용 계정 1개당 유료 라이선스 0원
채택 아래 Q&A 참고 — 결국 무료 개인 Gmail 계정으로 결정

왜 회사 정식 계정을 포기하고 무료 계정을 택했는가

회사 계정이 더 좋아 보이는데, 왜 안 썼나?

회사 정식 계정(Workspace)을 쓰려면 "온보딩 업무 전용" 신규 계정을 회사 차원에서 새로 만들어줘야 합니다. 그런데 이 프로젝트는 회사가 공식적으로 지시한 게 아니라 담당자가 업무 편의를 위해 개인적으로 진행 중인 비공식 프로젝트라, IT 부서에 신규 계정 발급을 정식으로 요청할 근거 자체가 없습니다. 즉 "회사 계정을 쓰고 싶어도, 새로 만들어달라고 할 수 없는" 구조적 한계가 있습니다.

그럼 실존하는 담당자 개인의 회사 계정을 쓰면 되지 않나?

가능하긴 하지만, 그 경우 Sheet와 Apps Script의 소유권이 그 개인에게 귀속됩니다. 나중에 그 담당자가 퇴사하면 계정이 비활성화되어 더 이상 활용할 수 없고, 자동화 스크립트도 함께 멈춥니다.

권한을 다른 사람에게 위임(delegate)하면 해결되지 않나?

Gmail 자체의 "권한 위임" 기능은 실제로 존재하지만, 검토해보니 두 가지 한계가 있었습니다.

① 위임받는 사람도 결국 자기 소유의 정식 라이선스 계정이 있어야 합니다. 위임은 같은 조직 내에서, 이미 정식 계정을 가진 사람에게만 가능합니다. 즉 "계정이 없는 사람에게 공짜로 권한을 나눠주는" 개념이 아니라서, 비용 문제가 근본적으로 해결되지 않습니다.

② Apps Script 자동화는 위임과 무관하게 배포한 계정에 실행 권한이 고정됩니다. 자동 실행되는 스크립트(웹앱)는 "누가 배포했는가"에 실행 주체가 영구히 묶이고, 이건 Gmail의 권한 위임 기능이 관여하는 영역이 아닙니다. 그 계정이 비활성화되면 스크립트가 멈추고, 새 담당자가 자기 계정으로 직접 재배포해야만 복구됩니다.

결국 회사 정식 계정이든 개인 회사 계정이든, 특정 개인에게 소유권이 귀속되는 근본 리스크는 동일하게 남습니다. 그렇다면 같은 리스크를 안고 갈 바에야, 비용이라도 들지 않는 무료 계정이 합리적이라고 판단했습니다.

무료 계정을 택하면서 감수한 한계

이중 관리 창구
이 프로젝트 전용 Gmail 계정은 회사 공식 메일 체계 밖에 있어, 문의·알림이 두 군데로 나뉘게 됩니다.
완화: 이 계정은 로그인용으로만 쓰고, 입사자와의 실제 소통(안내 메일 등)은 계속 회사 공식 메일 계정으로만 진행합니다.
메일 자동 발송 불가
Gmail 계정으로 자동 발송하면 "보낸사람"이 @gmail.com으로 노출되어, 신규 입사자에게 신뢰도 문제를 줄 수 있습니다.
완화: 안내 메일 자동 발송 기능 자체를 채택하지 않고, GAS는 데이터 조회·저장만 전담합니다. 메일은 계속 담당자가 회사 계정으로 수동 발송합니다.
